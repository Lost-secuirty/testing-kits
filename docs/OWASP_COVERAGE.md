# OWASP 2025 Coverage & Findings Export

Two pure-stdlib tooling utilities (zero runtime dependencies, advisory — they do not
gate a merge by themselves):

- [`tools/owasp_coverage.py`](../tools/owasp_coverage.py) — maps every `security/` and
  `ai/` harness to OWASP 2025 web (A01–A10) / OWASP LLM 2025 (LLMxx) categories and CWEs,
  and emits the matrix below.
- [`tools/findings_export.py`](../tools/findings_export.py) — normalizes harness findings
  into **SARIF 2.1.0** (for GitHub code-scanning and other dashboards) or flat JSON.

## Coverage matrix

> Snapshot generated by `python tools/owasp_coverage.py`. Regenerate after adding or
> renaming a security/ai harness. The `--self-test` gate (below) fails if a harness is
> unmapped or a mapping points at a harness that no longer exists, so the curated
> `REGISTRY` cannot silently drift from the tree.

| OWASP 2025 | Category | Harnesses | # |
|---|---|---|---|
| A01 | Broken Access Control | appsec, authz, session | 3 |
| A02 | Security Misconfiguration | security, misconfig | 2 |
| A03 | Software Supply Chain Failures | supplychain, supplychain_depth, ci_workflow_hardening, diff_secret_gate | 4 |
| A04 | Cryptographic Failures | jwt, pii_redaction, crypto | 3 |
| A05 | Injection | appsec, security, upload, advanced_injection | 4 |
| A06 | Insecure Design | rate_limit | 1 |
| A07 | Authentication Failures | appsec, authz, jwt, security, session | 5 |
| A08 | Software or Data Integrity Failures | appsec, upload, ci_workflow_hardening, cwe_kev_regression | 4 |
| A09 | Security Logging and Alerting Failures | pii_redaction, security_logging | 2 |
| A10 | Mishandling of Exceptional Conditions | exceptional_conditions | 1 |

LLM Top 10 (2025) categories covered: LLM01, LLM02, LLM05, LLM06, LLM08, LLM09, LLM10

(`cross` entries — `ast_sast`, `secure_codegen_eval`, `prompt_ab`, `agent_eval`,
`agent_memory_context`, `agentic`, `drift_detection` — are cross-cutting eval/oracle
tooling and intentionally not pinned to a single category.)

> LLM category note: tags follow each harness's own 2025 self-declaration. The 2025 OWASP
> LLM Top 10 renumbered earlier drafts — e.g. `ai/insecure_output_handling` is **LLM05**
> (Improper Output Handling) and `ai/sensitive_disclosure` is **LLM02** (Sensitive
> Information Disclosure). A unittest pins those two against regressing to old codes.

## Commands

```bash
python tools/owasp_coverage.py                # print the matrix above
python tools/owasp_coverage.py --self-test    # A01–A10 coverage + registry/tree sync
python tools/owasp_coverage.py --list-scenarios

python tools/findings_export.py --self-test   # SARIF/JSON normalization scenarios
python tools/findings_export.py --demo        # print a sample SARIF document

make coverage   # owasp_coverage self-test
make sarif      # findings_export self-test
```

Both are exercised by `unittest` (`tests/tools/test_owasp_coverage.py`,
`tests/tools/test_findings_export.py`) so they run in the standard `make test` lane.

## Scope & limits

- Coverage means **a harness exists for the category**, not that the category is fully
  exhausted. "No detected planted-class weakness" is not "secure."
- `findings_export` is a normalizer/exporter: it accepts dict or simple-object findings
  exposing `rule_id`/`check_name`, `severity`, `message`/`description`, and optional
  `cwe`/`file`/`line`. It does not itself run harnesses.
